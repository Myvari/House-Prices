# House Prices - Advanced Regression Techniques
### კონკურსის მიმოხილვა

ამ Kaggle-ის კონკურსის მიზანია სახლების გაყიდვის ფასის (SalePrice) პროგნოზირება. მონაცემები შეიცავს ბევრ მახასიათებლებს, როგორიცაა სახლის ზომა, ხარისხი, მდებარეობა და სხვა.
ამოცანა წარმოადგენს რეგრესიის პრობლემას, სადაც საჭიროა უწყვეტი რიცხვითი მნიშვნელობის პროგნოზირება და არა კატეგორიის განსაზღვრა.

ამოცანა ფასდება შემდეგი მეტრიკით:
RMSE(log(predicted SalePrice), log(actual SalePrice))

შეცდომა ითვლება ლოგარითმულ სივრცეში, ძვირი და იაფი სახლების პროგნოზირების შეცდომები უფრო დაბალანსებულად ფასდება

აქედან, მოდელი უნდა იყოს ოპტიმიზირებული არა პირდაპირ ფასებზე, არამედ მათ ლოგარითმებზე, ამ მიზეზით, პროექტში განსაკუთრებული ყურადღება დავუთმე log-transformed სამიზნე ცვლადზე გაწვრთნილ მოდელებს.

## მონაცემთა გაწმენდა და დამუშავება (Cleaning & Preprocessing)

**1. გამოტოვებული მნიშვნელობების (NA) ანალიზი**

პირველ ეტაპზე განხორციელდა მონაცემებში არსებული გამოტოვებული მნიშვნელობების დეტალური ანალიზი. თითოეული სვეტისთვის დაითვალა NA მნიშვნელობების პროცენტული წილი და ვიზუალიზაციის საშუალებით გამოიკვეთა ყველაზე პრობლემური სვეტები.

<img width="987" height="588" alt="image" src="https://github.com/user-attachments/assets/9aceff17-3992-4d12-a9c2-a66a23dbd20c" />

მნიშვნელოვანია აღინიშნოს, რომ ამ მონაცემებში NA მნიშვნელობები ყოველთვის არ ნიშნავს „დაკარგულ მონაცემს“. ბევრ შემთხვევაში ისინი თავად ატარებენ მნიშვნელოვან ინფორმაციას.

მაგალითად:

* PoolQC — NA ნიშნავს, რომ სახლს საერთოდ არ აქვს აუზი
- FireplaceQu — NA ნიშნავს, რომ ბუხარი არ არსებობს
+ Garage.. სვეტები — NA მიუთითებს, რომ სახლს არ აქვს ავტოფარეხი

ამ შემთხვევებში NA უნდა განვიხილოთ როგორც ცალკე კატეგორია („None“) და არა როგორც უბრალოდ გამოტოვებული მნიშვნელობა.

**2. მაღალი NA შემცველობის მქონე სვეტების დამუშავება**

სვეტები, სადაც NA მნიშვნელობების რაოდენობა ძალიან მაღალი იყო, შეფასდა ორი კრიტერიუმით:

* შეიცავს თუ არა სვეტი რეალურ ინფორმაციას
+ აქვს თუ არა გავლენა სამიზნე ცვლადზე (SalePrice):
  
<img width="804" height="117" alt="image" src="https://github.com/user-attachments/assets/9196588a-6621-41c2-8e88-34484b8e55a6" />


*წაშლილი სვეტები*:

Id — წარმოადგენს მხოლოდ უნიკალურ იდენტიფიკატორს და არ შეიცავს პროგნოზირებად ინფორმაციას
MiscFeature და MiscVal — ძალიან sparse (უმეტესად NA) და არასტაბილური გავლენა ფასზე
Alley, Fence — მაღალი NA + სუსტი კავშირი სამიზნესთან

ასეთი სვეტების დატოვება ზრდის ხმაურს და ამცირებს მოდელის სტაბილურობას.

**3. კატეგორიული სვეტების NA მნიშვნელობების შევსება**

კატეგორიულ სვეტებში გამოყენებული იქნა ორი ტიპის მიდგომა:

*(a) NA როგორც ინფორმაცია*

სვეტებისთვის, სადაც NA ნიშნავს ობიექტის არარსებობას გამოვიყენე:
```NA -> "None"```
რაც საშუალებას აძლევს მოდელს ისწავლოს, რომ კონკრეტული მახასიათებლის არარსებობა გავლენას ახდენს ფასზე.

*(b) რეალური გამოტოვებული მნიშვნელობები*

სვეტებისთვის, სადაც NA უბრალოდ დაკარგული მონაცემია შევავსეთ ყველაზე ხშირი მნიშვნელობით (mode).

**4. რიცხვითი სვეტების NA მნიშვნელობების შევსება**

რიცხვითი სვეტები დამუშავდა მათი მნიშვნელობის მიხედვით:

*(a) NA ნიშნავს არარსებობას -> 0*

*(b) რეალური დაკარგული მონაცემები -> მედიანა*

### Feature Engineering
Feature engineering-ის ეტაპზე შეიქმნა ახალი მახასიათებლები და გადამუშავდა არსებული სვეტები, რათა მოდელს მიეწოდოს უფრო ინფორმაციული და სტრუქტურირებული მონაცემები.

**1. ობიექტის არსებობის ინდიკატორები (Presence Features)**

  * HasGarage
  - HasPool
  - HasFireplace
  + HasMasVnr
    
**2. ფართობების აგრეგაცია**

ინდივიდუალური ფართობების სვეტები გაერთიანდა უფრო მაღალი დონის მახასიათებლებში:
  * TotalSF = TotalBsmtSF + 1stFlrSF + 2ndFlrSF


**3. კატეგორიული სვეტების დამუშავება**

კატეგორიული სვეტები გარდაიქმნა რიცხვით ფორმატში:
გამოყენებულია OneHotEncoder - თითოეული კატეგორია გადაიქცა ბინარულ სვეტად (0/1)

---

### Feature Selection
Feature selection-ის ეტაპზე მიზანი იყო ისეთი მახასიათებლების შერჩევა, რომლებიც მაქსიმალურად აუმჯობესებენ მოდელის პროგნოზირების უნარს და ამავდროულად ამცირებენ ზედმეტ სირთულეს.

**1. RFE (Recursive Feature Elimination)**
გამოყენებულ იქნა Recursive Feature Elimination (RFE) მეთოდი, რომლის მიზანია ნაკლებად მნიშვნელოვანი მახასიათებლების ეტაპობრივად მოცილება. ამორჩეული იქნა 20 ყველაზე მნიშვნელოვანი ცვლადი.
მნიშვნელოვანი ნაბიჯია რადგან ეს პროცესი უზრუნველყოფს, რომ მოდელში დარჩეს მხოლოდ რეალურად გავლენიანი კატეგორიები. (OHE-ს დროს შექმნილი ბევრი feature იწმინდება)

**2. კორელაციის ფილტრაცია (Correlation Filtering)** 
მონაცემებში გამოვლინდა ძლიერი კორელაცია გარკვეულ მახასიათებლებს შორის, როგორიცაა GarageArea - GarageCars, TotalBsmtSF - 1stFlrSF და სხვა მრავალი:

<img width="871" height="787" alt="image" src="https://github.com/user-attachments/assets/c699e2b7-3652-45ea-a9bc-86172545785d" />

გამოყენებულ იქნა კორელაციის ფილტრი (threshold ≈ 0.8) რომელიც საუკეთესო შედეგებს დებდა სხვადასხვა მნიშვნელობების გამოცდის შემდეგ. დაბალი ფილტრი უფრო ბევრი ცვლადის წაშლას გამოიწვევდა, რაც Underfitting-ის ალბათობას ზრდის, ხოლო მაღალი ფილტრი პირიქით, overfitting-ს გამოიწვევს (მოდელი იქნება კომპლექსური და ზედმეტად დაიწავლის ტრენინგის pattern-ებს)

### Pipeline-ის მიმდევრობა

Pipeline მოიცავს შემდეგ ნაბიჯებს:

+ 1.  Feature Engineering 
    
    ახალი მახასიათებლების დამატება (მაგ. HasGarage, TotalSF და სხვ.)
+ 2. Data Cleaning + ColumnTransformer (Preprocessing)

  ზედმეტი სვეტების მოცილება,
  რიცხვითი სვეტების შევსება (median),
  კატეგორიული სვეტების შევსება (None ან mode),
  OneHotEncoding
+ 3. Correlation Filter
     
  ძლიერ კორელირებული სვეტების მოცილება (redundancy-ის შემცირება)
+ 4. Feature Selection, RFE
     
  ყველაზე მნიშვნელოვანი feature-ების შერჩევა

  -```criterion```

  + ```test_rmse``` ≈ 64,000–65,000
  + ```test_r2``` ≈ 0.44
    
  მოდელი ვერ სწავლობს მონაცემებს, ხდება underfitting.
  
- **საშუალო სიღრმის ხეები (max_depth = 4–6)**
  + ```test_rmse``` ≈ 52,000–54,000
  + ```test_r2``` ≈ 0.61–0.64
  + მცირე ```rmse_gap```

  საუკეთესო შედეგები მიღებულია ამ დიაპაზონში, მოდელი ინარჩუნებს ბალანსს bias-სა და variance-ს შორის.

- **ღრმა ხეები (max_depth = 10 ან None)**
  + ```train_rmse``` ≈ 40,000
  + ```val_rmse``` ≈ 48,000
  + ```rmse_gap``` მნიშვნელოვნად იზრდება

  მიუთითებს overfitting-ზე, მოდელი ზედმეტად ერგება train მონაცემებს და ვერ გენერალიზდება.

ლოგ-ტრანსფორმირებული სამიზნე ცვლადის შემთხვევაში rmse_gap ძალიან მცირეა და მოდელიც უფრო სტაბილურია,თუმცა საბოლოო RMSE მნიშვნელობა მაინც შედარებით მაღალი რჩება.

საბოლოო ჯამში Decision Tree უკეთ ახერხებს არაწრფივი დამოკიდებულებების დაჭერას, ამცირებს underfitting-ს Linear Regression-თან შედარებით, თუმცა საბოლოო RMSE მნიშვნელობა არ უმჯობესდება, ღრმა ხეებში შეინიშნება overfitting
ანუ ზოგ შემთხვევაში Linear Regression უკეთ გენერალიზდება.

მიუხედავად იმისა, რომ Decision Tree უფრო მოქნილი მოდელია, ის არ უზრუნველყოფს უკეთეს გენერალიზაციას მოცემულ მონაცემებზე.

*(ამასთან Dataset-ის მცირე ზომა + ხმაური და ხეების დამახასიათებელი მარტივი overfitting-ი, იმის გათვალისწინებით რომ Linear Regression უფრო სტაბილურია შესაძლოა იწვევდეს ამ შედეგებს)*

### Random Forest

Random Forest მოდელი გამოყენებულ იქნა Decision Tree-ის გაუმჯობესებული ვერსიის სახით, რომლის მიზანია overfitting-ის შემცირება მრავალი ხის აგრეგაციის საშუალებით.

ტესტირება ჩატარდა სხვადასხვა პარამეტრებით:
  -```n_estimators```
  -```max_depth```
  -```min_samples_split```
  -```min_samples_leaf```

[Result Table](https://dagshub.com/myvari/House-Prices.mlflow/#/experiments/8/runs?searchFilter=&orderByKey=metrics.%60test_rmse%60&orderByAsc=true&startTime=ALL&lifecycleFilter=Active&modelVersionFilter=All+Runs&datasetsFilter=W10%3D)


- **საუკეთესო მოდელებმა აჩვენეს:**

  + ```test_rmse``` ≈ 51,000 – 52,000
  + ```test_r2``` ≈ 0.64 – 0.65

  Decision Tree-თან შედარებით შედეგები უფრო სტაბილურია

- ```rmse_gap``` ≈ 1,000 – 5,000
  
განსხვავება train და validation შედეგებს შორის შემცირებულია, Random Forest ამცირებს overfitting-ს Decision Tree-თან შედარებით

- **პარამეტრების გავლენა**
  + მცირე სიღრმე (max_depth = 3) -> underfitting
  + საშუალო სიღრმე (5–10) -> საუკეთესო შედეგები
  + ძალიან დიდი სიღრმე -> მცირე overfitting

ოპტიმალური შედეგები მიიღება საშუალო სირთულის მოდელებში.

ლოგ-ტრანსფორმირებული სამიზნე ცვლადის შემთხვევაში მოდელი უფრო სტაბილურია, თუმცა საბოლოო RMSE მნიშვნელობა არ უმჯობესდება მნიშვნელოვნად.

მიუხედავად იმისა, რომ Random Forest ამცირებს variance-ს, ის ვერ უზრუნველყოფს უკეთეს bias-variance ბალანსს მოცემულ ამოცანაში.

### Boosted Tree (XGBoost)

XGBoost გამოყენებულ იქნა როგორც ძლიერი მოდელი, რომელიც აერთიანებს მრავალი decision tree-ის შედეგებს boosting-ის პრინციპით.

ტესტირება ჩატარდა სხვადასხვა პარამეტრებით:
  -```n_estimators```
  -```learning_rate```
  -```max_depth```
  -```subsample```
  -```colsample_bytree```

- **საუკეთესო მოდელმა აჩვენა:**
  + ```test_rmse``` ≈ 48,700
  + ```test_r2``` ≈ 0.69
    
  ეს წარმოადგენს პროექტში მიღებულ ყველაზე დაბალ შეცდომას.

- **ლოგ-ტრანსფორმაციის გავლენა**

  ლოგ-ტრანსფორმირებული სამიზნე ცვლადის გამოყენებამ მნიშვნელოვნად გააუმჯობესა შედეგები:
    + ყველა საუკეთესო მოდელი მიღებულია log-target ვერსიაში
    + RMSE მნიშვნელობა მნიშვნელოვნად შემცირდა
  
  მიზეზი არის მონაცემების ასიმეტრიული განაწილება:
    + ამცირებს დიდი მნიშვნელობების გავლენას
    + სტაბილურობას მატებს მოდელს
    + ამარტივებს დამოკიდებულებების სწავლას

  *(დიდი ალბათობით log-მა დაასტაბილურა გრადიენტები და უკეთესი შედეგები ამან გამოიწვია)*

შეინიშნება ზომიერი overfitting:
  - ```rmse_gap``` ≈ 4,000 – 6,000

თუმცა მოდელი მაინც ინარჩუნებს კარგ გენერალიზაციას, შეცდომა მნიშვნელოვნად დაბალია სხვა მოდელებთან შედარებით.

- **პარამეტრების გავლენა:**
  + მცირე სიღრმის ხეები (max_depth = 3–4) -> საუკეთესო შედეგები
  + მაღალი სიღრმე -> overfitting
  + დაბალი learning rate + მეტი estimator -> უკეთესი შედეგები
